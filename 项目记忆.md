# 项目记忆

## 当前状态

- 当前仓库已从“工程底座”推进到“最小可玩原型”
- 已实现地图事件选择 -> 进入交互战斗 -> 战斗结算回写 -> 回合推进 -> 强制袭击 -> 撤离写入存档 的闭环
- 首屏已从按钮列表升级为地图事件板，支持图标、hover 提示、右侧事件面板实时联动
- 已补“家园整备与出发 -> 地图局内推进 -> 回归/死亡结算”显式闭环
- 战斗已具备最小交互：选敌、3 技能位、结束回合、使用道具、敌方依次反击、胜负回写、战果卡确认返回
- 已新增数值编辑台与全内容库编辑器，可在运行时调参与管理内置/自定义道具、敌人、英雄
- GitHub Pages 已上线，Web 版当前可直接访问试玩

## 已确认的关键决策

- 顶层状态固定为 `ContentDB / ProgressionState / RunState`
- 战斗是事件执行器，不是整局唯一主控制器（包括普通战斗、强制战斗、撤离战）
- 叙事事件大多数属于固定事件线
- 固定事件不会被随机刷新覆盖
- 局内战利品和局外永久存档必须分层
- `story_flags` 和 `unlock_flags` 默认允许跨死亡保留
- 运行版本约束更新为 `Godot 4.6`
- 地图事件板现在是首屏主入口，不再是左侧竖排按钮列表
- Web 平台需要跳过桌面窗口控制逻辑
- 当前主线体验重点不是扩大战斗系统，而是把“任务骨架推动”和“搜打撤分层结算”更直观地展示出来
- `gpt-5.3-codex` 对接当前 `newapi` 时，实际可用链路是 `responses`，不是 `chat/completions`

## 已知坑点

- 不要把“事件怎么出现”和“事件怎么结算”合并成一个枚举
- 不要默认用“三选一”理解回合事件板
- 不要把旧版长文档继续当成每次任务的完整输入
- 强制事件触发不占回合，不能被实现为普通随机事件替代
- 从 Godot 编辑器内直接运行时，编辑器会记住试玩窗口尺寸，可能覆盖运行时窗口调整
- 首屏地图 UI 的主要风险不再是逻辑，而是窗口尺寸、布局自适应与 Web 实机表现
- 后续新增 `DisplayServer / Window` 逻辑时，必须先考虑 `web` 平台分支
- 内容编辑器当前已能增删改查内置/自定义英雄、敌人、道具，但“把新内容挂入具体 battle / 掉落表 / 事件引用”还没做完
- 数值编辑台与内容库编辑器的信息密度已经提高，但仍有继续压缩滚动和提升检索效率的空间
- 交互战斗的打击表现已从直线闪现升级到弧线推进，但整体战斗演出仍然是轻量级表现层

## 当前建议优先级

1. 巩固地图事件板与首屏布局，让桌面/Web 两端都稳定可看
2. 继续提升交互战斗可读性与手感，而不是扩复杂系统
3. 巩固数值编辑台与全内容库编辑器的实用性
4. 视体验再决定是否继续扩 battle 引用编辑、掉落编辑和内容挂载能力
