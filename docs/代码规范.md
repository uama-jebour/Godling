# 代码规范

## GDScript 约定

- 类名、资源名、场景名保持一致性
- 文件名使用 `snake_case`
- 常量使用 `UPPER_SNAKE_CASE`
- 不把大型业务逻辑塞进单个控制器脚本

## 状态与职责

- `ContentDB` 只管内容
- `ProgressionState` 只管跨局持久化
- `RunState` 只管当前单局
- 不允许多个层重复维护同一份真相状态

## AutoLoad 规则

- 默认只使用既定 3 个顶层状态
- 新增 AutoLoad 前，先证明现有层级无法承载

## UI 规则

- UI 负责展示和输入，不直接写底层业务
- UI 不直接操作战斗单位、奖励发放或存档落盘
- UI 需要通过中间层或结构化结果对象更新

## 配置优先

- 新事件优先加配置，不优先写分支脚本
- 新任务优先改 JSON，不优先改逻辑硬编码
- 新掉落优先改掉落表，不优先改战斗脚本

## 验证策略

- 小改动（仅样式、文案、轻量布局或局部非行为改动）不要求每次执行全量冒烟回归
- 验证默认按影响面最小化：优先执行相关模块测试 + 最小手动验证
- 涉及主流程、数据契约、状态写入或跨模块行为变化时，仍需执行对应回归测试

## 实施前理解摘要

每次开始代码实现前，先用 5 条总结：

1. 本次目标
2. 本次范围外内容
3. 相关脚本和文档
4. 关键限制
5. 验收方式

若这 5 条仍有明显缺口，先修正理解，再动手。
